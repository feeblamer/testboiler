# Ошибки и несоответствия

## Этап проекта

Текущее состояние бойлерплейта: **ранний прототип / черновой MVP**, но не готовый бойлерплейт.

Причины:
- базовая структура уже создана;
- есть CLI-заготовка, шаблоны тестов и `sitecustomize.py`; - sitecustomize будет работать ? 
- при этом описание и реализация заметно расходятся;
- часть заявленных возможностей отсутствует или не доведена до рабочего состояния.

## Найденные проблемы

1. **Несоответствие имени пакета и CLI**
- В `description.md` везде используется `quickboiler`.
- В репозитории пакет называется `testboiler` и лежит в `testboiler/__main__.py`.
- В `pyproject.toml` нет `console_scripts` для `quickboiler`.
- Следствие: описанная команда `quickboiler ...` фактически не настроена.

2. **Отсутствует часть заявленной структуры**
- В описании заявлены `src/` и пакет `quickboiler/`.
- В репозитории нет `src/`, вместо `quickboiler/` используется `testboiler/`.
- Структура проекта не соответствует собственной спецификации.

3. **Не реализована команда `dockerize`** - РЕАЛИЗОВЫАТЬ НЕ НАДО
- В `description.md` команда есть.
- В `testboiler/__main__.py` она не добавлена.
- Следствие: одна из основных возможностей существует только в тексте.

4. **Игнорируется конфиг `framework`**
- В `quickboiler.cfg` есть блок:
  - `pytest: true`
  - `unittest: true`
- В `testboiler/__main__.py` команда `run` всегда запускает оба набора тестов.
- Следствие: переключение фреймворков, заявленное в описании, не работает.

5. **`sitecustomize.py` зависит от `yaml`, но зависимость не гарантирована**
- В `sitecustomize.py` есть `import yaml`.
- В `pyproject.toml` список `dependencies` пустой.
- В `requirements.txt` нет `pyyaml`.
- Следствие: автозагрузка через `sitecustomize.py` может упасть ещё до выполнения тестов.

6. **Конфиг по умолчанию невалиден для реального запуска**
- В `quickboiler.cfg` указано `library: <lib==2.0.1>`. - это шаблон на то он и шаблон 
- Это выглядит как плейсхолдер, а не рабочее значение.
- При попытке использовать такой пакет установка завершится ошибкой.

7. **Тесты захардкожены под `requests` и не связаны с конфигом** - нужно примеры а не реальный код или просто комментарий как создать тесты
- В обоих тестах импортируется только `requests`. - это для примера было в шаблоне не должно быть библиотек 
- Если в `quickboiler.cfg` указать другую библиотеку, шаблонные тесты перестанут соответствовать цели проекта.
- Следствие: проект не является универсальным бойлерплейтом "для любой Python-библиотеки".

8. **`requirements.txt` не заполняется автоматически** - нужно поправить в шаблоне можно указать pytest
- В описании сказано, что файл "будет заполнен автоматически".
- В репозитории этого механизма нет.
- Сейчас файл просто содержит фиксированные строки.

9. **`main.py` не связан с CLI и выглядит лишним** - нужно убрать это создано uv автоматически
- В `main.py` только вывод `Hello from testboiler!`.
- Он не участвует в описанном пользовательском сценарии.
- Это признак незавершённой или сгенерированной заготовки.

10. **README почти пустой** - позже можно будет заполнить
- `README.md` не описывает установку, команды, ограничения и фактическое состояние проекта.
- Для бойлерплейта это серьёзный недочёт, потому что входная документация должна быть короче и точнее, чем `description.md`.

11. **Заявление о "готовности к расширению" преждевременно**
- Docker, CI, multiple libraries и другие расширения перечислены как направление, но не реализованы. - не ужно докера вообще 
- На текущем этапе это дорожная карта, а не функциональность.

## Вывод

Проект находится на этапе **идеи, частично оформленной в код как демонстрационный прототип**.

До полноценного MVP не хватает как минимум:
- привести имя проекта, пакет и документацию к одному названию;
- починить зависимости (`pyyaml`, entry points, рабочий запуск);
- связать конфиг с реальным поведением CLI;
- убрать захардкоженный `requests` из шаблонов;
- либо реализовать заявленные команды, либо убрать их из описания.
